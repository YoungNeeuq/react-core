🇻🇳 Tiếng Việt
📖 Định nghĩa
Virtual DOM (VDOM) là một bản sao JavaScript object của cây DOM thật, được React duy trì trong bộ nhớ. Khi state/props thay đổi, React tạo VDOM mới, so sánh (diff) với VDOM cũ, rồi chỉ cập nhật những phần thực sự thay đổi lên DOM thật — gọi là Reconciliation.
State thay đổi
→
Render → VDOM mới
→
Diff VDOM cũ vs mới
→
Commit minimal changes → DOM
🔍 Thuật toán Diffing — O(n) với 2 heuristic
DOM diffing thực chất là O(n³) — React tối ưu xuống O(n) bằng 2 giả định:

1. Same type → Update, Different type → Replace toàn bộ
<div> → <div>: update attributes. <div> → <span>: unmount div, mount span mới (bỏ toàn bộ subtree).

2. Key prop để track list items
Không có key: React diff theo index → dễ sai. Có key: React track đúng element dù vị trí thay đổi.
⚠️ Hiểu lầm phổ biến
Virtual DOM không phải lúc nào cũng nhanh hơn direct DOM manipulation. Nó trade off: thêm memory overhead + diffing cost để đổi lấy sự tiện lợi (developer experience) và đảm bảo correctness. Svelte không dùng VDOM mà vẫn rất nhanh.

1
Virtual DOM là gì? Tại sao React cần nó?
Cơ bản ▼
Virtual DOM là một JS object tree đại diện cho cấu trúc UI. React dùng nó vì:

1. Abstraction: dev viết JSX → React lo việc cập nhật DOM (cross-platform: web, native, VR)
2. Batching: gom nhiều state update thành 1 lần render
3. Correctness: React đảm bảo UI luôn phản ánh đúng state hiện tại

Điểm nâng cao: VDOM không phải về tốc độ — Svelte không có VDOM vẫn nhanh hơn. VDOM là về developer experience và cross-platform abstraction.
2
Tại sao không nên dùng index làm key trong list? Khi nào thì được?
Trung bình ▼
Khi dùng index làm key và list bị thêm/xóa/reorder:
— React nhìn key vẫn là 0,1,2,3... → nghĩ là element cũ, chỉ update props
— Kết quả: state nội bộ (input value, animation) bị giữ nguyên sai chỗ

Ví dụ lỗi: List todo có input. Xóa item đầu tiên → React không unmount component cũ, chỉ update text. Input value bị gán sai item.

Khi nào index OK: List static hoặc chỉ append cuối, không reorder, không xóa giữa, items không có state nội bộ (chỉ display data).
3
Reconciliation xảy ra thế nào khi component type thay đổi?
Nâng cao ▼
Khi React thấy element cũ và mới khác type (ví dụ <div> → <section>, hoặc <UserCard> → <AdminCard>):

1. Unmount toàn bộ subtree cũ — cleanup tất cả effects, destroy state
2. Mount subtree mới từ đầu — tạo DOM mới, chạy lại effects

Practical implication: Đừng định nghĩa component bên trong component. Mỗi render tạo function reference mới → React nghĩ là type khác → unmount/remount liên tục → mất state, flicker.
🇬🇧 English
📖 Definition
The Virtual DOM is a JavaScript object tree kept in memory that mirrors the real DOM. When state/props change, React creates a new VDOM, diffs it against the previous one, and commits only the minimal necessary changes to the real DOM. This process is called Reconciliation.
🔍 Diffing Algorithm — O(n) with 2 heuristics
Naïve tree diffing is O(n³). React achieves O(n) with two assumptions:

1. Same type → Update, Different type → Replace entire subtree
<div> → <div>: update attributes. <div> → <span>: unmount div, mount new span (destroys entire subtree).

2. key prop tracks list items across renders
Without keys: React diffs by index → incorrect behavior. With keys: React correctly identifies elements even when position changes.

What is the Virtual DOM? Why does React need it?
Basic ▼
The VDOM is a lightweight JS object tree representing the UI. React uses it for:

1. Abstraction: devs write JSX, React handles cross-platform rendering (web, native, VR)
2. Batching: groups multiple state updates into a single render pass
3. Correctness: guarantees UI always reflects current state

Advanced point: VDOM isn't about raw speed — Svelte is faster without it. It's about developer experience and cross-platform abstraction.

Medium ▼
When using index as key and the list is reordered/filtered/prepended:
— Keys remain 0,1,2,3... → React thinks same elements, only updates props
— Internal state (input values, animations) stays attached to the wrong item

Index is acceptable when: the list is static or append-only, never reorders, never deletes from middle, and items have no internal state (pure display).

React Fiber Architecture
Senior level 2 câu phỏng vấn
▼
🇻🇳 Tiếng Việt
📖 Fiber là gì?
React Fiber (React 16+) là kiến trúc lại hoàn toàn của reconciler. Trước Fiber (Stack Reconciler): reconciliation là synchronous và không thể interrupt — cây lớn → block UI thread hàng trăm ms.

Fiber biến reconciliation thành linked list của "units of work", mỗi unit có thể pause, resume, abort, hoặc reuse. Đây là nền tảng của Concurrent Mode.
🔍 2 Phase của Fiber
Phase 1 — Render (Reconciliation, interruptible):
React duyệt cây Fiber, tính toán changes, tạo "work-in-progress" tree. Có thể bị interrupt bởi higher-priority update.

Phase 2 — Commit (Synchronous, uninterruptible):
Apply changes vào DOM thật. Chạy useLayoutEffect. Không thể interrupt — phải hoàn thành trong 1 lần.
⚠️ Tại sao commit phase không thể interrupt?
Nếu interrupt giữa chừng khi đang ghi vào DOM → UI ở trạng thái inconsistent (một nửa cũ, một nửa mới). Người dùng thấy layout bị vỡ. Chỉ có render phase mới được interrupt vì nó không chạm DOM.
🎯 Câu hỏi phỏng vấn
1
React Fiber khác gì Stack Reconciler cũ? Tại sao cần thay đổi?
Senior ▼
Stack Reconciler (React 15 trở về trước):
— Dùng call stack của JS — synchronous, không interrupt được
— Cây 10,000 nodes có thể block UI 200ms+ → janky animations

Fiber Reconciler (React 16+):
— Mỗi component là 1 Fiber node trong linked list
— Reconciliation chia thành units of work, dùng requestIdleCallback-like scheduler
— Có thể pause khi có animation frame, resume khi idle

Kết quả thực tế: Concurrent Mode, Suspense, useTransition, useDeferredValue đều dựa vào khả năng interrupt này.
2
Render phase và Commit phase khác nhau thế nào? Tại sao quan trọng?
Senior ▼
Render Phase (a.k.a. Reconciliation):
— Tính toán changes, không có side effects, pure
— Có thể chạy nhiều lần (Strict Mode cố tình double-invoke để phát hiện impure renders)
— Interruptible — React có thể bỏ và bắt đầu lại

Commit Phase:
— Apply changes vào DOM thật, synchronous
— Chạy useLayoutEffect cleanup + setup đồng bộ
— Sau đó schedule useEffect asynchronously

Thực tế quan trọng: Đây là lý do render function phải pure — nó có thể bị gọi nhiều lần. Side effects (fetch, DOM mutation) phải nằm trong useEffect/commit phase.
🇬🇧 English
📖 What is Fiber?
React Fiber (React 16+) is a complete rewrite of the reconciler. The old Stack Reconciler was synchronous and uninterruptible — a large tree could block the UI thread for hundreds of milliseconds.

Fiber transforms reconciliation into a linked list of "units of work" that can pause, resume, abort, or reuse. This is the foundation for Concurrent Mode.
🔍 Two phases
Phase 1 — Render (interruptible): React traverses the Fiber tree, computes changes, builds a work-in-progress tree. Can be interrupted by higher-priority updates.

Phase 2 — Commit (synchronous, uninterruptible): Applies changes to the real DOM. Runs useLayoutEffect. Must complete atomically to avoid inconsistent UI.

Senior ▼

📖 Cơ chế hoạt động
React lưu state trong Fiber node của component (không phải trong closure). Mỗi useState call tương ứng 1 slot trong linked list của Fiber. Đây là lý do hooks không được gọi trong điều kiện/loop — thứ tự phải nhất quán giữa các render để React biết slot nào là của hook nào. 
// React nội bộ (simplified)
let currentHook = null

function useState(initialValue) {
  const hook = getOrCreateHook() // lấy từ Fiber linked list
  if (hook.state === undefined) hook.state = initialValue
  const setState = (newVal) => {
    hook.state = typeof newVal === 'function' ? newVal(hook.state) : newVal
    scheduleRender() // trigger re-render
  }
  return [hook.state, setState]
}

1
setState trong React có phải synchronous không? Giải thích.
Hay hỏi ▼
Không. setState là asynchronous và batched.

Khi gọi setState, React không re-render ngay. Nó schedule một re-render. Nếu gọi nhiều setState trong cùng một event handler → React batch lại thành 1 lần render duy nhất.

React 18+: Automatic batching — batch cả trong async callbacks (setTimeout, Promise.then). Trước React 18 chỉ batch trong React event handlers.

const handleClick = () => {
  setCount(c => c + 1)
  setCount(c => c + 1) // React 18: 2 lần, batch thành 1 render
  console.log(count) // vẫn là giá trị cũ! State chưa update
}
Functional update setState(prev => prev + 1) luôn an toàn hơn setState(count + 1) vì nó dùng giá trị thực tế mới nhất trong queue, không bị stale closure.
2
Stale closure là gì? Cho ví dụ thực tế và cách fix.
Senior ▼
Stale closure xảy ra khi hàm capture giá trị biến từ render cũ và không cập nhật khi state thay đổi.

function Counter() {
  const [count, setCount] = useState(0)

  useEffect(() => {
    const id = setInterval(() => {
      console.log(count) // 🔴 luôn log 0 — stale closure!
      setCount(count + 1) // 🔴 luôn set về 1
    }, 1000)
    return () => clearInterval(id)
  }, []) // [] — effect chạy 1 lần, capture count=0 mãi mãi
}
Fix 1 — Functional update (tốt nhất khi không cần đọc count):
setCount(prev => prev + 1)

Fix 2 — Thêm count vào dependency array (effect chạy lại mỗi khi count đổi):
useEffect(() => { ... }, [count])

Fix 3 — useRef để lưu giá trị mới nhất:
const countRef = useRef(count); countRef.current = count;
3
Tại sao hooks không được gọi bên trong if/loop?
Trung bình ▼
React xác định hook nào tương ứng state nào bằng thứ tự gọi (index trong linked list của Fiber), không phải tên biến.

Nếu hook bị skip trong 1 render (do điều kiện), thứ tự thay đổi → React gán sai state cho sai hook → bugs không thể đoán trước.

// ❌ SAI
if (condition) {
  const [a] = useState(0) // slot 1 khi condition=true
  // slot 1 bị skip khi condition=false
}
const [b] = useState(0) // slot 2 hoặc 1 tùy condition → bug!

React có re-render khi setState cùng giá trị không?
Trung bình ▼
React dùng Object.is để so sánh. Nếu giá trị mới === giá trị cũ → bail out, không re-render.

Tuy nhiên: React vẫn có thể render component một lần để kiểm tra trước khi bail out (tùy vị trí trong Fiber tree). Điều này không vi phạm gì nếu render function là pure.

Trap với object/array: setState({...obj}) tạo reference mới → Object.is trả false → luôn re-render dù nội dung giống hệt. Đây là lý do nên immutable update.
🇬🇧 English
📖 Internal mechanism
React stores state in the component's Fiber node (not in the closure). Each useState call corresponds to a slot in the Fiber's hook linked list. This is why hooks can't be called conditionally — the order must be consistent between renders so React knows which slot belongs to which hook.
🎯 Interview Questions
1
Is setState synchronous? Explain batching in React 18.
Common ▼
No. setState is asynchronous and batched. Calling it schedules a re-render — calling it multiple times in an event handler produces a single render pass.

React 18 Automatic Batching: batches updates inside async callbacks (setTimeout, Promise.then) too. React 17 and below only batched inside React event handlers.

Always use functional updates setState(prev => prev + 1) when the new state depends on the previous — avoids stale closure bugs in batched scenarios.
2
What is a stale closure in React? Real-world example and fix.
Senior ▼
A stale closure occurs when a function captures a variable value from a past render and never sees the updated value.

Classic example: setInterval inside useEffect([]) captures the initial state value and never updates it.

Fixes:
1. Functional update: setCount(prev => prev + 1)
2. Add to dependency array: useEffect(() => {...}, [count])
3. Store latest in ref: const ref = useRef(count); useEffect(() => { ref.current = count })

🇻🇳 Tiếng Việt
📖 useEffect chạy khi nào?
useEffect chạy sau khi browser paint (asynchronous). So sánh:

— useEffect: sau paint → không block visual update, tốt cho fetch/subscriptions
— useLayoutEffect: sau DOM mutation nhưng trước paint → dùng khi cần đọc/ghi DOM layout (scroll position, measurements) để tránh flicker
🔍 Dependency Array — 3 behavior
useEffect(fn) — không có array: chạy sau mỗi render
useEffect(fn, []) — array rỗng: chạy 1 lần sau mount
useEffect(fn, [a, b]) — có deps: chạy lại khi a hoặc b thay đổi (so sánh bằng Object.is)
⚠️ Cleanup function — cực quan trọng
Cleanup chạy trước mỗi lần effect chạy lại VÀ khi component unmount. Nếu không cleanup:
— Event listener vẫn attached sau unmount → memory leak + bug
— Fetch request completed nhưng component đã unmount → setState trên unmounted component (warning)
— Subscription vẫn active → multiple subscriptions chồng nhau 
function SearchResults({ query }) {
  useEffect(() => {
    let ignore = false // race condition fix
    const controller = new AbortController()

    fetch(`/api/search?q=${query}`, { signal: controller.signal })
      .then(r => r.json())
      .then(data => {
        if (!ignore) setResults(data) // bỏ qua nếu đã cleanup
      })

    return () => {
      ignore = true
      controller.abort() // cancel in-flight request
    }
  }, [query])
}

2
Race condition trong useEffect khi fetch là gì? Cách fix?
Senior ▼
Khi dependency thay đổi nhanh (user gõ search), nhiều fetch requests được gửi. Request trước có thể return sau request mới → UI hiện data sai.

Fix 1 — ignore flag (xem code ở trên): Mỗi effect tạo biến ignore=false. Cleanup set ignore=true. Trong callback, chỉ setState nếu !ignore.

Fix 2 — AbortController: Cancel request cũ khi effect cleanup. Cả hai thường dùng kết hợp.

Fix tốt nhất: Dùng React Query / SWR — chúng handle race condition tự động.
3
Tại sao React 18 Strict Mode chạy effect 2 lần trong development?
Hay gặp ▼
React 18 Strict Mode intentionally mount → unmount → mount lại mỗi component trong dev (không production) để:

1. Phát hiện side effects trong render phase (render chạy 2 lần)
2. Phát hiện thiếu cleanup trong useEffect
3. Chuẩn bị cho Concurrent Mode — React có thể unmount/remount component để optimize

Nếu app bị bug khi Strict Mode chạy effect 2 lần → đó là bug thực sự trong code, không phải Strict Mode lỗi. Fix bằng cách viết cleanup đúng.
4
Khi nào KHÔNG nên dùng useEffect?
Thực chiến ▼
React team (Dan Abramov, docs mới) liệt kê nhiều trường hợp dùng useEffect sai:

❌ Transform data for rendering: Tính toán derived state từ props/state — làm trực tiếp trong render, không cần effect.
❌ Handle user events: Logic xử lý click/submit → event handler, không phải effect.
❌ Fetch on mount mà không cleanup: Dùng React Query/SWR thay thế.
❌ Sync state giữa 2 state variables: Dấu hiệu cần refactor thành 1 state hoặc derived value.

Nhớ: useEffect để sync React state với hệ thống bên ngoài React (DOM, browser APIs, 3rd party libraries). Nếu logic chỉ liên quan đến React state/props → không cần effect.
🇬🇧 English
📖 When does useEffect run?
useEffect runs after the browser paints (asynchronous). Contrast:
— useEffect: after paint → doesn't block visual update; good for fetch/subscriptions
— useLayoutEffect: after DOM mutations but before paint → use when reading/writing layout to avoid flicker
🎯 Interview Questions
1
What is a race condition in useEffect? How do you fix it?
Senior ▼
When a dependency changes rapidly (e.g., search input), multiple fetch requests are in flight. An older request might resolve after a newer one → UI shows stale data.

Fix 1 — ignore flag: Each effect creates let ignore = false. Cleanup sets ignore = true. Only call setState if !ignore.
Fix 2 — AbortController: Cancel in-flight requests during cleanup.
Best fix: Use React Query / SWR — they handle this automatically with built-in deduplication and cancellation.
2
When should you NOT use useEffect?
Practical ▼
Per React's own documentation:

❌ Deriving/transforming state for rendering — do it directly during render
❌ Handling user events — use event handlers
❌ Fetching on mount without cleanup — use React Query/SWR
❌ Syncing two state variables — sign you need to refactor to single source of truth

Mental model: useEffect is for synchronizing React with external systems (DOM, browser APIs, third-party libs). If the logic only involves React state/props → no effect needed.

useMemo vs useCallback — khi nào thực sự cần?
Hay hỏi nhất 4 câu phỏng vấn
▼
🇻🇳 Tiếng Việt
📖 Cơ chế
useMemo(fn, deps): cache return value của fn. Chỉ recompute khi deps thay đổi.
useCallback(fn, deps): cache function reference. Chỉ tạo function mới khi deps thay đổi.

Mối quan hệ: useCallback(fn, deps) tương đương useMemo(() => fn, deps) 
// useMemo — cache computed value
const filteredList = useMemo(() =>
  items.filter(i => i.active).sort((a,b) => a.name > b.name ? 1 : -1)
, [items])

// useCallback — cache function reference cho React.memo child
const handleDelete = useCallback((id) => {
  setItems(prev => prev.filter(i => i.id !== id))
}, []) // empty deps — function không capture variable nào thay đổi
🎯 Câu hỏi phỏng vấn
1
Sự khác nhau giữa useMemo và useCallback? (câu kinh điển)
Kinh điển ▼
useMemo memoize giá trị trả về (return value của callback).
useCallback memoize chính function đó (function reference).

useCallback(fn, deps) chính xác tương đương useMemo(() => fn, deps)

Khi nào dùng cái nào:
— useMemo: tính toán nặng (filter/sort array lớn, regex, mathematical computation)
— useCallback: function truyền xuống component có React.memo, hoặc function là dependency trong useEffect/useMemo khác
2
Khi nào useMemo/useCallback KHÔNG có ích và thậm chí gây hại?
Senior ▼
Không có ích khi:
— Computation rất nhanh (primitive math, string concat) — overhead của useMemo lớn hơn lợi ích
— Deps thay đổi mỗi render (object/array mới tạo inline) — cache hit rate 0%
— Component không dùng React.memo — useCallback vô nghĩa vì parent re-render → child re-render anyway

Gây hại khi:
— Thêm code complexity, khó đọc
— Deps array sai → stale memoized value → bugs khó debug
— Tốn memory lưu cached value + deps array

Rule: Đo trước (React DevTools Profiler), optimize sau. Đừng memo mọi thứ.
3
Cho ví dụ cụ thể khi useCallback thực sự cần thiết.
Thực chiến ▼ 
// ✅ useCallback thực sự cần
const ParentList = () => {
  const [items, setItems] = useState([...])
  const [filter, setFilter] = useState('')

  // Không có useCallback: handleDelete mới mỗi render
  // → MemoRow luôn re-render dù items không đổi
  const handleDelete = useCallback((id) => {
    setItems(p => p.filter(i => i.id !== id))
  }, []) // setItems stable, không cần trong deps

  return items.map(item =>
    <MemoRow key={item.id} item={item} onDelete={handleDelete} />
  )
}

// MemoRow chỉ re-render khi item hoặc onDelete thay đổi
const MemoRow = React.memo(({ item, onDelete }) => { ... })
Khi filter state thay đổi → ParentList re-render → nếu không có useCallback, handleDelete là function mới → MemoRow re-render dù item không đổi. Với useCallback → MemoRow skip re-render.
4
useMemo có đảm bảo không tính lại không? (trick question)
Trick ▼
Không! React có thể discard cached value bất cứ lúc nào — ví dụ khi memory pressure cao, khi React cần free memory cho Concurrent features.

Điều này có nghĩa:
— Không dùng useMemo như nguồn truth — chỉ dùng như optimization hint
— Không lưu side effects trong useMemo (mutating external state, fetch) — nó có thể re-run bất ngờ
— Trong tương lai React Forget (React compiler) sẽ tự động memo — useMemo manual sẽ ít cần hơn

React docs chính thức: "You should only rely on useMemo as a performance optimization, not as a semantic guarantee."
🇬🇧 English
📖 Mechanism
useMemo(fn, deps): caches the return value of fn. Recomputes only when deps change.
useCallback(fn, deps): caches the function reference itself. Creates new function only when deps change.

Relationship: useCallback(fn, deps) is equivalent to useMemo(() => fn, deps)
🎯 Interview Questions
1
Difference between useMemo and useCallback?
Classic ▼
useMemo memoizes the return value; useCallback memoizes the function reference.
useCallback(fn, deps) ≡ useMemo(() => fn, deps)

Use useMemo for expensive computations (filter/sort large arrays). Use useCallback for callbacks passed to React.memo children or used as useEffect/useMemo dependencies.
2
Does useMemo guarantee the value is never recomputed? (trick)
Trick ▼
No! React may discard cached values at any time (e.g., under memory pressure or for Concurrent features). useMemo is an optimization hint, not a semantic guarantee. Never put side effects inside useMemo — it could re-run unexpectedly.

Khi nào component re-render? 4 nguyên nhân
Hay hỏi nhất 5 câu phỏng vấn
▼
🇻🇳 Tiếng Việt
📖 4 nguyên nhân gây re-render
1. State thay đổi: useState / useReducer setter được gọi
2. Props thay đổi: Parent re-render → child nhận props mới (kể cả giá trị giống hệt nhưng reference khác)
3. Context thay đổi: Bất kỳ consumer nào của context sẽ re-render khi context value đổi
4. Parent re-render: Quan trọng! Mặc định, khi parent re-render, tất cả children đều re-render — kể cả khi props không đổi (nếu không dùng React.memo)
⚠️ Hiểu lầm nguy hiểm
"React chỉ re-render khi props thay đổi" — SAI.
React re-render children khi parent re-render, bất kể props có thay đổi không. React.memo mới là thứ ngăn điều này.
🎯 Câu hỏi phỏng vấn
1
React.memo hoạt động thế nào? Có phải so sánh deep không?
Hay hỏi ▼
React.memo(Component) wrap component, so sánh props cũ và mới trước khi re-render. Nếu props giống nhau → skip re-render, trả component cũ.

Mặc định: shallow comparison — so sánh từng prop bằng Object.is. Với primitive (string, number, boolean) thì đúng. Với object/array/function: so sánh reference — nếu parent tạo mới mỗi render thì luôn khác.

Custom comparison:
React.memo(Component, (prevProps, nextProps) => {
  // return true = skip re-render (same)
  // return false = re-render (different)
  return prevProps.id === nextProps.id
})
Cạm bẫy phổ biến: <MemoComp onClick={() => doSomething()} /> — arrow function mới mỗi render → React.memo luôn re-render. Cần useCallback.
2
Tại sao truyền object literal vào props luôn gây re-render dù dùng React.memo?
Thực chiến ▼

<MemoChild config={{ timeout: 3000 }} />
<MemoChild onClick={() => handle()} />

// ✅ Fix: hoist ra ngoài component hoặc dùng useMemo/useCallback
const STYLE = { color: 'red' } // constant, same reference
<MemoChild style={STYLE} />
Object.is so sánh reference, không so sánh nội dung. {color:'red'} === {color:'red'} → false. Mỗi render tạo object mới → reference mới → React.memo thấy "props thay đổi" → re-render.

Context API gây re-render toàn bộ consumer — cách tối ưu?
Senior ▼
Context re-renders tất cả consumers khi value thay đổi, không quan tâm component đó có dùng phần vừa đổi không.

Giải pháp 1 — Tách context: Thay vì 1 context khổng lồ, tách thành nhiều context nhỏ. Component chỉ subscribe context mà nó thực sự cần.

Giải pháp 2 — use-context-selector: Library cho phép select phần cụ thể của context, chỉ re-render khi phần đó thay đổi (như Redux selector).

Giải pháp 3 — Zustand / Jotai: Có selector built-in, chỉ re-render khi selected value thay đổi.

5
Children pattern có giúp tránh re-render không?
Advanced ▼
Có! Đây là pattern ít người biết nhưng rất hiệu quả.

// ❌ Slow: SlowComponent re-render mỗi khi count đổi
function Parent() {
  const [count, setCount] = useState(0)
  return (<div>
    <button onClick={() => setCount(c=>c+1)}>{count}</button>
    <SlowComponent /> // re-render mỗi lần
  </div>)
}

// ✅ Fast: children prop không bị recreate khi count đổi
function Counter({ children }) { // chỉ re-render Counter
  const [count, setCount] = useState(0)
  return <div><button onClick={() => setCount(c=>c+1)}>{count}</button>{children}</div>
}
function App() {
  return <Counter><SlowComponent /></Counter>
  // SlowComponent tạo trong App scope, không re-render theo Counter
}
🇬🇧 English
📖 4 causes of re-renders
1. State changes: useState/useReducer setter called
2. Props change: Parent re-renders → child gets new props (even if value is identical but reference differs)
3. Context changes: Every consumer re-renders when context value changes
4. Parent re-renders: By default, when a parent re-renders, all children re-render regardless of props — only React.memo prevents this

How does React.memo work? Is the comparison deep or shallow?
Common ▼
React.memo wraps a component and compares previous and next props before re-rendering. If props are the same → skip re-render.

Default: shallow comparison — compares each prop with Object.is. Works for primitives. For objects/arrays/functions: compares references — fails if parent creates new objects/functions on every render.

Khi nào tách ra custom hook:
— Logic lặp lại ở nhiều component (useFetch, useLocalStorage, useDebounce)
— Component quá phức tạp, cần tách concern
— Muốn test logic độc lập với component
— Encapsulate third-party library API

  }, [value, delay])
  return debounced
}

📖 Cơ chế

// 1. Tạo context với default value
const ThemeContext = createContext({ theme: 'light', toggle: () => {} })

// 2. Provider — wrap ở cấp cao nhất cần share
function ThemeProvider({ children }) {
  const [theme, setTheme] = useState('light')
  const value = useMemo(() => ({
    theme,
    toggle: () => setTheme(t => t === 'light' ? 'dark' : 'light')
  }), [theme]) // ⚠️ useMemo value để tránh tạo object mới mỗi render
  return <ThemeContext.Provider value={value}>{children}</ThemeContext.Provider>
}

// 3. Consumer — ở bất kỳ cấp nào bên dưới Provider

  const { theme, toggle } = useContext(ThemeContext)
  return <button onClick={toggle}>{theme}</button>

// 1. useLocalStorage — sync state với localStorage
function useLocalStorage(key, initial) {
  const [value, setValue] = useState(() => {
    try {
      const stored = localStorage.getItem(key)
      return stored ? JSON.parse(stored) : initial
    } catch { return initial }
  })
  const set = useCallback((v) => {
    setValue(v)
    localStorage.setItem(key, JSON.stringify(v))
  }, [key])
  return [value, set]
}
// 2. useDebounce — delay value update
function useDebounce(value, delay = 300) {
  const [debounced, set] = useState(value)
  useEffect(() => {
    const id = setTimeout(() => set(value), delay)
    return () => clearTimeout(id)
  }, [value, delay])
  return debounced
}
// 3. usePrevious — lấy giá trị render trước
function usePrevious(value) {
  const ref = useRef()
  useEffect(() => { ref.current = value }) // no deps — mỗi render đều update
  return ref.current // trả về giá trị của render TRƯỚC khi effect chạy
}
// 4. useIntersectionObserver — lazy load / infinite scroll
function useInView(options = {}) {
  const ref = useRef(null)
  const [inView, setInView] = useState(false)
  useEffect(() => {
    const obs = new IntersectionObserver(
      ([entry]) => setInView(entry.isIntersecting), options
    )
    if (ref.current) obs.observe(ref.current)
    return () => obs.disconnect()
  }, [])
  return [ref, inView]
}
🎯 Câu hỏi phỏng vấn
1
Viết custom hook useDebounce. Giải thích tại sao cần cleanup.
Coding ▼
Xem code trên. Điểm quan trọng cần giải thích:

Tại sao cần cleanup: Mỗi lần value thay đổi, effect chạy lại → setTimeout mới được tạo. Cleanup (clearTimeout) cancel setTimeout cũ chưa fire. Nếu không cleanup → nhiều timeout cùng chạy → debounce không hoạt động → state update nhiều lần.

Tại sao lazy init useState: useState(() => ...) thay vì useState(...) → initializer function chỉ chạy 1 lần khi mount, không mỗi render. Quan trọng khi initial value cần tính toán (đọc localStorage).
2
usePrevious hoạt động thế nào? Tại sao ref.current trả về giá trị cũ?
Tricky ▼
Thứ tự xảy ra trong 1 render cycle:

1. Component render — ref.current đọc lúc này → vẫn là giá trị chưa update (previous)
2. DOM update (commit)
3. useEffect chạy → ref.current = value → lưu giá trị mới

Lần render tiếp theo lại lặp lại: đọc ref.current ở bước 1 → lúc đó là "previous" của render này.

Không có deps array — effect chạy sau mỗi render để đảm bảo luôn sync. Dùng ref thay vì state vì không muốn trigger thêm render.

How does usePrevious work? Why does ref.current return the old value?
Tricky ▼
Timing within one render cycle:
1. Component renders — reads ref.current → still previous value
2. DOM commits
3. useEffect runs → ref.current = value → stores new value

Next render: step 1 reads ref again → it's now the "previous" for this render. No deps array means the effect syncs after every render. Uses ref instead of state to avoid triggering additional renders.

📖 3 Nguyên tắc cốt lõi của Redux
1. Single Source of Truth: Toàn bộ state của app nằm trong một store duy nhất — dễ debug, serialize, hydrate.

🔍 Các khái niệm quan trọng
Action: Plain object { type: 'INCREMENT', payload: 1 }. type là string bắt buộc. payload là convention (Flux Standard Action).

Action Creator: Function trả về action object. Giúp centralize action creation, dễ test, type-safe.

Reducer: Pure function xử lý action → trả state mới. Không được: mutate state, call API, gọi Date.now()/Math.random() (impure).

Store: Object chứa state, expose getState(), dispatch(), subscribe().

Selector: Function state => derivedValue. Dùng reselect để memoize — tránh recompute khi state không liên quan thay đổi.

Middleware: Function hook vào dispatch pipeline. Nhận store → next → action. Dùng cho: logging, async, analytics. 
// Redux "vanilla" — để hiểu bản chất
// Action types
const INCREMENT = 'counter/INCREMENT'
const DECREMENT = 'counter/DECREMENT'

// Action creators
const increment = (amount) => ({ type: INCREMENT, payload: amount })

// Reducer — pure function, NO mutations
function counterReducer(state = { value: 0 }, action) {
  switch (action.type) {
    case INCREMENT:
      return { ...state, value: state.value + action.payload }
    case DECREMENT:
      return { ...state, value: state.value - action.payload }
    default:
      return state // LUÔN return state mặc định
  }
}

// Memoized selector với reselect
const selectTotal = createSelector(
  [(state) => state.items],
  (items) => items.reduce((sum, i) => sum + i.price, 0)
  // chỉ recompute khi items thay đổi

Tại sao Reducer phải là pure function? Điều gì xảy ra nếu mutate state trực tiếp?
Nền tảng ▼
Tại sao pure:
— Predictability: Cùng state + action → luôn cùng kết quả. Dễ test, dễ reason.
— Time-travel debugging: DevTools replay actions từ đầu → phải pure mới replay đúng.
— Memoization: useSelector so sánh reference — reducer trả object mới → UI biết cần update.

Nếu mutate trực tiếp:

  return state  // trả cùng reference
}

2
Redux middleware là gì? Viết custom middleware đơn giản.
Senior ▼
Middleware là higher-order function hook vào dispatch pipeline. Có thể intercept, log, modify hoặc delay action trước khi đến reducer.

// Middleware signature: store => next => action => result
const loggerMiddleware = (store) => (next) => (action) => {
  console.log('Dispatching:', action)
  console.log('Before state:', store.getState())
  const result = next(action) // pass to next middleware / reducer
  console.log('After state:', store.getState())
  return result
}

// Redux Thunk — middleware đơn giản nhất cho async
const thunkMiddleware = (store) => (next) => (action) => {
  if (typeof action === 'function') { // action là function → thunk
    return action(store.dispatch, store.getState)
  }
  return next(action) // action bình thường → pass tiếp
}
3
Selector là gì? Tại sao cần memoize selector?
Quan trọng ▼
Selector là function đọc và transform data từ Redux store. useSelector(selector) chạy selector mỗi khi store update và so sánh kết quả bằng === để quyết định re-render.

Vấn đề khi không memoize:
// ❌ Tạo array mới mỗi lần — luôn re-render
const activeUsers = useSelector(state =>
  state.users.filter(u => u.active) // new array reference mỗi lần
)

// ✅ createSelector — memoize, chỉ recompute khi users thay đổi
const selectActiveUsers = createSelector(
  [state => state.users],
  users => users.filter(u => u.active)
)
const activeUsers = useSelector(selectActiveUsers)
createSelector (từ reselect, built into RTK): cache kết quả, chỉ recompute khi input selectors trả giá trị mới.

Why must reducers be pure functions? What breaks if you mutate state?
Foundation ▼
Pure reducers enable predictability, time-travel debugging, and memoized selectors.

Redux Thunk vs Redux Saga — key differences?
Senior ▼
Thunk: Actions can be functions. Low learning curve, uses async/await. Hard to cancel or manage races.
Saga: Generator-based, declarative effects (take/put/call/race). High learning curve but powerful: built-in cancellation, takeLatest, race conditions, complex orchestration.

Use Thunk for simple async. Use Saga for complex workflows. RTK recommends createAsyncThunk for most cases.

📖 RTK là gì? Tại sao cần?
Redux Toolkit là official Redux package giải quyết các phàn nàn phổ biến với Redux vanilla:
— Quá nhiều boilerplate (action types, action creators, reducers riêng biệt)
— Cần cài thêm nhiều packages (immer, reselect, thunk)
— Không biết cấu trúc folder "đúng"

// createSlice — gom action types + action creators + reducer vào 1 chỗ
import { createSlice, createAsyncThunk } from '@reduxjs/toolkit'

// Async thunk — tự tạo pending/fulfilled/rejected action types
export const fetchUser = createAsyncThunk(
  'users/fetchById',
  async (userId, { rejectWithValue }) => {
    try {
      const res = await fetch(`/api/users/${userId}`)
      if (!res.ok) throw new Error('Failed')
      return await res.json()
    } catch (err) {
      return rejectWithValue(err.message)
    }
  }
)

const usersSlice = createSlice({
  name: 'users',
  initialState: { data: [], status: 'idle', error: null },
  reducers: {
    // Immer cho phép "mutate" — thực ra tạo immutable copy bên dưới
    addUser(state, action) {
      state.data.push(action.payload) // ✅ OK với RTK/Immer
    },
    removeUser(state, action) {
      state.data = state.data.filter(u => u.id !== action.payload)
    },
  },
  extraReducers: (builder) => { // handle async thunk actions
    builder
      .addCase(fetchUser.pending,   (state) => { state.status = 'loading' })
      .addCase(fetchUser.fulfilled, (state, action) => {
        state.status = 'succeeded'
        state.data.push(action.payload)
      })
      .addCase(fetchUser.rejected, (state, action) => {
        state.status = 'failed'
        state.error = action.payload
      })
  }
})

export const { addUser, removeUser } = usersSlice.actions
export default usersSlice.reducer

const store = configureStore({

🎯 Câu hỏi phỏng vấn
1
RTK dùng Immer — tại sao "mutate" trong reducer lại OK?
Hay hỏi ▼
RTK wrap reducers với Immer. Immer dùng JavaScript Proxy để intercept mutations. Khi reducer "mutate" state, Immer không thực sự mutate — nó tạo một bản copy (draft), record tất cả changes, rồi produce một immutable object mới.

Lợi ích: Code ngắn hơn, ít spread operator hơn, dễ nested updates hơn. Underlying state vẫn immutable — Redux DevTools vẫn hoạt động.

Lưu ý: Chỉ "mutate" trong RTK reducers — không phải ở nơi khác. Nếu return value từ reducer, Immer dùng return value đó thay vì draft.
2
createAsyncThunk tự tạo những action types nào? extraReducers dùng để làm gì?
RTK ▼
createAsyncThunk('users/fetch', asyncFn) tự tạo 3 action types:
— users/fetch/pending: khi async function bắt đầu
— users/fetch/fulfilled: khi resolve, payload = return value
— users/fetch/rejected: khi throw hoặc rejectWithValue, payload = error

extraReducers: Xử lý actions tạo từ nơi khác (ngoài slice này) — thường là async thunk actions. Dùng builder API để type-safe.

Tại sao không viết trong reducers? Vì createSlice chỉ tạo actions từ tên keys của reducers. Async thunk actions không thuộc slice nào — phải handle trong extraReducers.

Redux DevTools time-travel hoạt động thế nào?
DevTools ▼
Redux DevTools lưu lại toàn bộ action history và state snapshot sau mỗi action. Time-travel là replay tất cả actions từ đầu đến action muốn inspect.

Hoạt động được vì:
— State là serializable plain objects → có thể serialize/deserialize
— Reducers là pure functions → replay cùng sequence actions → cùng state
— Không có side effects trong reducer → replay không có unintended effects

Tính năng mạnh: Jump to any state, import/export state, skip actions, test reducers với specific state snapshot.

Only "mutate" inside RTK reducers — not in selectors, components, or other places.

React Query: Independent of state management, works with any async function (REST, GraphQL, anything), more flexible cache key strategy, larger ecosystem.
